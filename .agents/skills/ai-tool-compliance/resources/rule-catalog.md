## P0 룰 상세

### AUTH-P0-001: 신규 가입자 Role=Guest 강제

**왜 필요한가**:
초대 기반 권한 통제 시스템의 기반이다. 신규 가입자에게 기본적으로 Guest 역할을 부여함으로써 최소 권한 원칙(Principle of Least Privilege)을 시스템 수준에서 강제한다. Guest가 아닌 역할은 반드시 관리자의 명시적 초대/승인을 통해서만 부여되어야 한다.

**위반 시 비즈니스 영향**:
- 누구나 가입 즉시 관리자 기능에 접근 가능 → 내부 데이터 유출, 비인가 설정 변경
- 악의적 사용자가 가입만으로 AI 모델 호출, 비용 발생 가능 → 비용 폭주
- 보안 사고 발생 시 "가입 시점에 왜 권한이 부여되었는가"에 대한 감사 추적 불가

**검증 방법 요약**:
1. `signup`, `register`, `createUser` 등 가입 로직이 포함된 파일을 Grep으로 탐지
2. 해당 파일에서 `role.*['"]guest['"]` 패턴 존재 여부 확인 (must_contain)
3. 가입 로직에서 `role.*['"]admin['"]` 또는 `role.*['"]manager['"]` 패턴 탐지 시 즉시 FAIL (must_not_contain)

**판정 기준**:
- PASS: 가입 코드에서 role='guest' 또는 role='pending'을 기본값으로 설정
- FAIL: 가입 시 admin/manager/viewer 역할이 기본 부여되는 코드 존재

**참고 가이드 섹션**: Section 2 (역할 정의), Section 3 (권한 매트릭스)

### AUTH-P0-002: Guest 메뉴/API 접근 차단

**왜 필요한가**:
Guest에게 툴명, AI 모델명, 내부 인프라 구조, 비용 정보, 시스템 아키텍처를 노출하면 사전 정찰(reconnaissance)에 활용될 수 있다. Guest는 명시적으로 허용된 메뉴와 API에만 접근할 수 있어야 한다.

**위반 시 비즈니스 영향**:
- 내부 시스템 구조 노출 → 공격 표면 확대
- AI 모델명/버전 노출 → 모델 취약점 표적화 가능
- 비용 구조 노출 → 경쟁사에 단가 정보 유출

**검증 방법 요약**:
1. 미들웨어/라우터 가드에서 Guest 역할에 대한 명시적 접근 차단 로직 탐지
2. `guest.*block`, `guest.*deny`, `role.*guest.*reject` 등의 패턴 존재 확인
3. 보호된 API 엔드포인트에 Guest 차단 미들웨어가 적용되었는지 확인

**판정 기준**:
- PASS: 미들웨어 또는 라우터에서 Guest 역할에 대한 명시적 접근 차단 존재
- FAIL: Guest 역할 체크 없이 모든 인증된 사용자에게 동일한 접근 허용

**참고 가이드 섹션**: Section 3 (권한 매트릭스), Section 4 (게임/메뉴별 접근 제어)

### AUTH-P0-003: 서버 최종 권한 검증

**왜 필요한가**:
클라이언트 사이드 권한 체크는 브라우저 DevTools, API 직접 호출, 프록시 변조 등으로 손쉽게 우회 가능하다. 서버에서 모든 API 요청에 대해 최종 권한 검증을 수행해야 실질적인 접근 통제가 보장된다.

**위반 시 비즈니스 영향**:
- 프론트엔드 권한 체크만 있을 경우 → API 직접 호출로 모든 보호 우회 가능
- 권한 없는 사용자가 관리자 기능 실행 → 데이터 변조, 삭제, 설정 변경
- 보안 감사 시 "서버 레벨 권한 검증 부재"는 중대 취약점으로 분류

**검증 방법 요약**:
1. API 라우터 정의 (`router.get`, `router.post`, `router.put`, `router.delete`) 탐지
2. 각 라우터 핸들러에 `auth`, `verify`, `authenticate` 등 권한 검증 미들웨어 존재 여부 확인
3. 권한 검증 미들웨어가 누락된 라우터 발견 시 FAIL

**판정 기준**:
- PASS: 모든 API 핸들러에 서버 사이드 권한 검증 미들웨어 적용
- FAIL: 일부 API에 미들웨어 누락, 또는 클라이언트 권한 체크만 존재

**참고 가이드 섹션**: Section 5 (서버 사이드 검증)

### SEC-P0-004: Firestore 직접 접근 금지

**왜 필요한가**:
클라이언트에서 Firestore를 직접 read/write하면 보안 규칙(Security Rules)에만 의존하게 되는데, 복잡한 비즈니스 로직을 Security Rules로 완벽히 표현하기 어렵다. Cloud Functions를 경유하면 서버 레벨에서 정교한 검증과 로깅이 가능하다.

**위반 시 비즈니스 영향**:
- Security Rules 설정 오류 시 → 전체 데이터베이스 노출 위험
- 클라이언트에서 직접 쓰기 시 → 데이터 무결성 검증 우회 가능
- 접근 로그가 남지 않음 → 보안 사고 시 원인 추적 불가
- Firestore 과다 읽기/쓰기 → 비용 폭주 (클라이언트 통제 불가)

**검증 방법 요약**:
1. 클라이언트 코드 경로(src/, client/, app/, pages/ 등)에서 Firestore SDK 직접 import 탐지
2. `firebase/firestore`, `getDocs`, `setDoc`, `addDoc`, `getFirestore`, `collection()` 등의 패턴을 must_not_contain으로 검사
3. functions/ 또는 server/ 경로는 검증 제외 (서버에서의 Firestore 접근은 허용)

**판정 기준**:
- PASS: 클라이언트 코드에 Firestore 직접 접근 코드 없음
- FAIL: 클라이언트 경로에서 Firestore SDK import 또는 직접 접근 코드 발견

**참고 가이드 섹션**: Section 7 (Firestore 접근 정책)

### SEC-P0-005: 외부 API Gateway 강제

**왜 필요한가**:
외부 AI API(Gemini, OpenAI, Claude 등)를 클라이언트 또는 서버에서 직접 호출하면 API 키 노출, 비용 추적 불가, 행동 로그 누락이 동시에 발생한다. 반드시 내부 Gateway(Cloud Functions)를 경유해야 키 관리, 비용 추적, 로깅, 캐싱을 중앙에서 통제할 수 있다.

**위반 시 비즈니스 영향**:
- API 키가 클라이언트에 포함 시 → 키 탈취 후 무제한 호출 → 수천만 원 비용 청구 가능
- 비용 추적 누락 → 월말에야 비용 폭주 인지 → 사후 대응만 가능
- 로그 누락 → 누가 어떤 모델을 얼마나 사용했는지 추적 불가
- 캐싱 미적용 → 동일 요청 반복 호출로 불필요한 비용 발생

**검증 방법 요약**:
1. 전체 코드에서 `fetch('https://...')` 패턴 중 localhost가 아닌 외부 호출 탐지
2. 특히 `openai.com`, `generativelanguage.googleapis.com`, `anthropic.com` 등 AI API 도메인 직접 호출 탐지
3. 클라이언트 경로에서의 외부 API 직접 호출은 무조건 FAIL

**판정 기준**:
- PASS: 모든 외부 AI API 호출이 내부 Gateway(Cloud Functions)를 경유
- FAIL: 클라이언트 또는 비-Gateway 서버에서 AI API 직접 호출 발견

**배점 근거**: 15점 (P0 최고 배점). Gateway 우회는 키 노출 + 비용 추적 불가 + 로그 누락을 동시에 유발하는 최고 위험 패턴.

**참고 가이드 섹션**: Section 6 (Gateway 원칙), Section 7 (인프라 접근 정책)

### SEC-P0-009: 민감 텍스트 서버 처리

**왜 필요한가**:
AI 프롬프트 원문과 응답 전문에는 비즈니스 로직, 사용자 데이터, 시스템 지시가 포함될 수 있다. 이를 클라이언트에 직접 전달하면 브라우저 캐시, 네트워크 스니핑, 클라이언트 로그 등을 통해 민감 정보가 유출된다.

**위반 시 비즈니스 영향**:
- 프롬프트 원문 노출 → 프롬프트 엔지니어링 전략 유출
- AI 응답 전문에 내부 데이터 포함 시 → 비인가 사용자에게 정보 유출
- 응답에 모델명/비용 포함 시 → Guest에게 인프라 정보 노출 (AUTH-P0-002 연관)
- 개인정보가 포함된 프롬프트가 클라이언트에 캐싱 → 개인정보보호법 위반

**검증 방법 요약**:
1. API 응답(`res.json`, `res.send`)에서 프롬프트/응답 원문 포함 여부 탐지
2. `res.json(.*password`, `console.log(.*token` 등 민감 정보 노출 패턴 검사
3. 응답에 참조 ID만 반환하는 패턴 확인

**판정 기준**:
- PASS: 클라이언트에는 응답 ID/요약만 반환, 원문은 서버에서만 처리/저장
- FAIL: AI 응답 전문 또는 프롬프트 원문이 클라이언트 응답에 직접 포함

**참고 가이드 섹션**: Section 8 (민감 데이터 처리)

### COST-P0-006: 모델 호출 비용 로그

**왜 필요한가**:
AI 모델 호출은 건당 비용이 발생하며, 모델/토큰 수에 따라 비용 편차가 크다. 호출별 비용을 기록하지 않으면 비용 폭주를 사전에 감지할 수 없고, 사후 분석도 불가능하다.

**위반 시 비즈니스 영향**:
- 비용 폭주 사전 감지 불가 → 월말 청구서에서야 문제 인지
- 팀/사용자별 비용 분석 불가 → 비용 최적화 방향 수립 불가
- 모델 선택 최적화 불가 → 고비용 모델 과다 사용 지속
- 예산 초과 시 원인 특정 불가 → "누가 얼마나 썼는지" 추적 불가

**필수 로그 필드**: `model`, `inputTokens`, `outputTokens`, `estimatedCost`, `userId`, `timestamp`

**검증 방법 요약**:
1. AI 모델 호출 코드(`openai`, `vertexai`, `gemini`, `anthropic` 등) 탐지
2. 해당 호출 경로에 `cost`, `token`, `usage`, `billing` 관련 로그 기록 존재 확인
3. 필수 4필드(model, inputTokens, outputTokens, estimatedCost) 모두 기록되는지 확인

**판정 기준**:
- PASS: 모든 AI 모델 호출 경로에 필수 필드를 포함한 비용 로그 기록
- FAIL: 일부 호출 경로에 로그 누락, 또는 필수 필드 미기록

**참고 가이드 섹션**: Section 9 (비용 추적)

### COST-P0-007: BQ 스캔 비용 로그

**왜 필요한가**:
BigQuery는 쿼리당 스캔 데이터량에 비례하여 과금된다. 하나의 비효율적 쿼리가 테라바이트를 스캔할 수 있으며, 이를 추적하지 않으면 예상치 못한 대규모 비용이 발생한다.

**위반 시 비즈니스 영향**:
- 비효율 쿼리 1건이 수십만 원 비용 발생 가능 → 감지 없이 반복 실행
- 쿼리 최적화 판단 근거 부재 → 파티셔닝/클러스터링 도입 시점 판단 불가
- 월간 BQ 비용 예측 불가 → 예산 초과 시 원인 특정 불가

**필수 로그 필드**: `bytesProcessed`, `estimatedCost`, `queryId`, `userId`

**검증 방법 요약**:
1. BigQuery 관련 코드(`bigquery`, `BigQuery`, `createQueryJob`) 탐지
2. 해당 코드에 `totalBytesProcessed`, `bytesProcessed`, `cost` 관련 로그 기록 확인
3. BQ 사용 코드가 없으면 N/A 판정 (패널티 없음)

**판정 기준**:
- PASS: 모든 BQ 쿼리 실행 경로에 bytesProcessed + estimatedCost 기록
- FAIL: BQ 사용 코드는 있으나 비용 로그 누락
- N/A: BQ 미사용 프로젝트

**참고 가이드 섹션**: Section 9 (비용 추적)

### COST-P0-011: 캐시 우선 조회

**왜 필요한가**:
동일하거나 유사한 요청에 대해 매번 AI API를 호출하면 불필요한 비용이 반복 발생한다. 캐시 레이어를 두어 이전 결과를 재활용하면 비용을 절감하고 응답 속도도 개선된다.

**위반 시 비즈니스 영향**:
- 동일 요청 반복 호출 → 불필요한 비용 누적 (호출량에 비례)
- 캐시 미적용 → 피크 시간대 API 호출 폭주 → Rate Limit 초과 장애
- 응답 지연 증가 → 사용자 경험 저하

**검증 방법 요약**:
1. AI API 호출 코드(`openai`, `vertexai`, `gemini`, `anthropic`) 탐지
2. 호출 전 `cache`, `Cache`, `redis`, `memo` 등 캐시 조회 로직 존재 확인
3. 캐시 미스 시에만 실제 API 호출이 실행되는 패턴 확인

**판정 기준**:
- PASS: 고비용 API 호출 전 캐시 조회 로직 존재 (Redis/Memcached/Firestore 캐시 등)
- FAIL: 매 요청마다 캐시 없이 AI API 직접 호출

**참고 가이드 섹션**: Section 10 (캐시 정책)

### LOG-P0-008: 실패 요청 로그 필수

**왜 필요한가**:
실패 요청(4xx, 5xx, timeout)의 로그가 누락되면 장애 원인 분석, 보안 사고 탐지, 패턴 기반 이상 징후 감지가 모두 불가능해진다. 성공 요청만 기록하는 것은 "잘된 것만 보는" 편향이다.

**위반 시 비즈니스 영향**:
- 장애 발생 시 → 원인 추적 불가 → MTTR(평균 복구 시간) 증가
- 반복 실패 패턴(무차별 대입 공격 등) → 보안 위협 감지 불가
- 외부 API 장애 → 실패 로그 없으면 SLA 협상/보상 청구 근거 부재
- 사용자 불만 원인 분석 불가 → CS 대응 지연

**검증 방법 요약**:
1. error handler, catch 블록, `onError` 패턴 탐지
2. 각 에러 처리 경로에 `logger`, `log.error`, `winston`, `pino` 등 로그 호출 존재 확인
3. 로그 호출이 누락된 catch 블록 발견 시 FAIL

**판정 기준**:
- PASS: 4xx/5xx/timeout 모든 실패 케이스에 로그 기록 (에러 코드, 스택 포함)
- FAIL: 성공 요청만 로그 기록, 실패 케이스 누락

**참고 가이드 섹션**: Section 11 (로그 정책)

---

### LOG-P0-010: 권한 변경 감사 로그

**왜 필요한가**:
역할 변경, 권한 부여/회수, 초대 발송 등은 시스템 보안에 직접 영향을 미치는 민감 이벤트이다. 이 이벤트의 전수 기록 없이는 "누가 언제 누구의 권한을 변경했는가"를 추적할 수 없다.

**위반 시 비즈니스 영향**:
- 비인가 권한 상승 발생 시 → 변경자/시점 추적 불가
- 내부 감사(audit) 시 → 권한 변경 이력 부재 → 감사 실패
- 보안 사고 발생 시 → "이 사용자의 권한이 언제 변경되었는가" 증명 불가
- 규제 준수 요구 시 → 감사 추적 부재는 규제 위반

**필수 이벤트**: role 변경, 권한 부여/회수, 초대 발송, 사용자 차단/해제

**필수 로그 필드**: `eventType`, `targetUserId`, `changedBy`, `oldRole`, `newRole`, `timestamp`

**검증 방법 요약**:
1. 권한 변경 코드(`updateRole`, `changeRole`, `setRole`) 탐지
2. 해당 코드에 `audit`, `auditLog`, `eventLog` 등 감사 로그 호출 존재 확인
3. 필수 필드(eventType, targetUserId, changedBy, oldRole, newRole) 기록 확인

**판정 기준**:
- PASS: 모든 권한 변경 이벤트에 필수 필드를 포함한 감사 로그 기록
- FAIL: 권한 변경 코드는 있으나 감사 로그 누락

**참고 가이드 섹션**: Section 12 (감사 로그)

## P1 룰 (권장 수준) — 4개

P1 룰은 위반 시 배포를 차단하지 않으나, 3건 이상 위반 시 Warning 등급이 부여되어 리뷰가 필수가 된다. P0 전수 통과 후 점진적으로 도입한다.

### P1-DOM-001: 허용 도메인 화이트리스트

**왜 필요한가**:
AI Gateway가 호출할 수 있는 외부 도메인을 제한하지 않으면, Gateway를 통해 임의의 외부 서비스를 호출하는 데 악용될 수 있다. 화이트리스트로 허용 도메인을 명시하면 SSRF(Server-Side Request Forgery) 위험을 차단한다.

**위반 시 비즈니스 영향**:
- SSRF 공격 → Gateway를 경유하여 내부 네트워크 접근 가능
- 비인가 외부 서비스 호출 → 데이터 유출 경로 확보
- API 키가 예상하지 못한 도메인에 전달 → 키 탈취 위험

**검증 방법 요약**: Gateway 설정 파일에서 allowedDomains/whitelist 설정 존재 여부 확인 (config_check)

**참고 가이드 섹션**: Section 6 (Gateway 원칙) 부속 항목

---

### P1-STAT-002: 사용자별/팀별 사용량 집계

**왜 필요한가**:
개별 호출 비용 로그(COST-P0-006)만으로는 "어떤 팀이 가장 많은 비용을 사용하는가", "비용 트렌드가 증가 추세인가"를 파악하기 어렵다. 일/주/월 단위 집계를 통해 비용 최적화 의사결정을 지원한다.

**위반 시 비즈니스 영향**:
- 팀별 비용 분담 불가 → 비용 귀속 갈등
- 사용량 트렌드 파악 불가 → 용량 계획(capacity planning) 실패
- 과다 사용 팀/사용자 식별 불가 → 비용 최적화 방향 수립 불가

**검증 방법 요약**: 사용량 집계 코드/대시보드 존재 여부 확인 (code_review)

**참고 가이드 섹션**: Section 9 (비용 추적) 부속 항목

---

### P1-COST-003: 예산 상한 알림

**왜 필요한가**:
비용 로그를 기록(COST-P0-006)하더라도 능동적 알림이 없으면 비용 초과를 사후에야 인지한다. 팀/프로젝트별 임계값을 설정하고 초과 시 즉시 알림(Slack/PagerDuty/Email)을 보내야 한다.

**위반 시 비즈니스 영향**:
- 비용 초과를 월말 청구서에서야 발견 → 이미 손실 발생
- 비정상적 사용(도용, 루프 버그) → 실시간 감지 불가로 피해 확대
- 예산 관리 실패 → CEO/CFO 레벨 에스컬레이션

**검증 방법 요약**: 예산 임계값 설정 + 알림 연동 코드/설정 확인 (config_check + code_review)

**참고 가이드 섹션**: Section 10 (비용 통제) 부속 항목

---

### P1-LOG-004: 로그 보관 정책

**왜 필요한가**:
로그를 무한정 보관하면 스토리지 비용이 누적되고, 보관 기간이 너무 짧으면 보안 감사 시 필요한 이력이 소실된다. 명확한 보관 정책(Hot/Cold/삭제)을 수립해야 비용과 규제 준수의 균형을 맞출 수 있다.

**위반 시 비즈니스 영향**:
- 보관 정책 부재 → Firestore 스토리지 비용 무한 증가
- 보관 기간 미설정 → 규제 감사 시 요구 기간 내 로그 부재 → 규제 위반
- 오래된 로그 미삭제 → 불필요한 개인정보 보유 → 개인정보보호법 위반 위험

**검증 방법 요약**: 로그 보관 설정(retention 정책, TTL, 아카이빙 설정) 존재 여부 확인 (config_check)

**권장 보관 기준**:
- Hot (Firestore): 90일
- Cold (BigQuery): 365일
- 이후: 삭제 또는 장기 아카이브

**참고 가이드 섹션**: Section 11 (로그 정책) 부속 항목
