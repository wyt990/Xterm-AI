# AI Tool Compliance - 점수 가중치 기준

> 내부 AI 툴 필수 구현 가이드 v1.1 기반 정량적 준수 점수 체계.
> 영역별 가중치 근거, 등급 기준, 커스텀 가중치 적용 방법을 포함한다.

---

## 영역별 가중치 배분

### 기본 가중치

| 영역 | 가중치 | 만점 | 포함 룰 | 룰 수 |
|------|--------|------|---------|-------|
| 보안 (Security) | **40%** | 40 | SEC-P0-004, SEC-P0-005, SEC-P0-009 | 3 |
| 권한 (Auth) | **25%** | 25 | AUTH-P0-001, AUTH-P0-002, AUTH-P0-003 | 3 |
| 비용 (Cost) | **20%** | 20 | COST-P0-006, COST-P0-007, COST-P0-011 | 3 |
| 로그 (Logging) | **15%** | 15 | LOG-P0-008, LOG-P0-010 | 2 |
| **합계** | **100%** | **100** | **11개 P0 룰** | **11** |

## 가중치 근거

### 보안 (Security) — 40%

**근거**: 보안 영역은 단일 위반이 가장 큰 비즈니스 피해를 유발한다.

1. **API 키 노출 위험 (SEC-P0-005)**: Gateway 미경유 시 API 키가 클라이언트에 포함될 수 있으며, 키 탈취 시 무제한 외부 API 호출로 수천만 원 비용 청구가 가능하다. 복구 비용과 피해 규모가 가장 크다.

2. **데이터 유출 위험 (SEC-P0-004)**: Firestore 직접 접근 허용 시 Security Rules 설정 오류 하나로 전체 데이터베이스가 노출될 수 있다. 고객 데이터 유출 시 법적 책임, 평판 손상, 사업 중단까지 이어진다.

3. **민감 정보 유출 (SEC-P0-009)**: 프롬프트/응답 원문이 클라이언트에 노출되면 비즈니스 로직, 내부 데이터, 프롬프트 전략이 유출된다. 경쟁사 또는 공격자에게 직접적인 이점을 제공한다.

4. **다른 영역과의 연쇄 효과**: 보안 위반은 비용(API 키 악용), 로그(접근 기록 부재), 권한(우회) 영역에 연쇄적으로 영향을 미친다. 따라서 보안에 가장 높은 가중치를 부여한다.

**결론**: 보안 위반은 피해 규모가 가장 크고, 복구 비용이 높으며, 연쇄 파급 효과를 가진다. 40%의 가중치는 이 위험도를 반영한다.

### 권한 (Auth) — 25%

**근거**: 권한은 "누가 무엇에 접근할 수 있는가"를 통제하는 핵심 메커니즘이다.

1. **최소 권한 원칙 (AUTH-P0-001)**: 가입 즉시 높은 권한이 부여되면 공격 표면이 극대화된다. Guest 강제는 전체 권한 체계의 기반이다.

2. **정보 노출 통제 (AUTH-P0-002)**: Guest에게 시스템 구조, 모델명, 비용 정보가 노출되면 보안의 사전 정찰(reconnaissance) 단계가 완료된다. 이는 보안 영역 위반의 전제 조건이 되므로 높은 가중치가 필요하다.

3. **서버 검증 (AUTH-P0-003)**: 클라이언트 권한 체크만으로는 우회가 가능하다. 서버 검증이 없으면 모든 권한 체계가 무력화된다.

**결론**: 권한은 보안의 기반 계층이지만, 보안 자체의 직접적 피해(키 노출, 데이터 유출)보다는 "접근 통제 실패로 인한 간접 위험"이므로 보안보다 낮은 25%를 부여한다.

### 비용 (Cost) — 20%

**근거**: AI 서비스의 특성상 비용은 운영의 지속 가능성에 직접 영향을 미친다.

1. **비용 가시성 (COST-P0-006)**: 모델별/호출별 비용을 추적하지 않으면 최적화 의사결정의 근거가 없다. "어떤 모델이 비용 대비 효율적인가"를 판단할 수 없다.

2. **BQ 비용 폭주 (COST-P0-007)**: BigQuery 쿼리 1건이 테라바이트를 스캔할 수 있으며, 추적 없이는 감지 불가능하다.

3. **캐시 미적용 (COST-P0-011)**: 동일 요청 반복 호출은 불필요한 비용의 가장 흔한 원인이다.

**결론**: 비용 관리 실패는 서비스 중단(예산 소진)이나 예산 초과로 이어지지만, 보안/권한 위반처럼 즉각적인 데이터 유출이나 시스템 침해를 유발하지는 않는다. 20%의 가중치는 운영 지속성 리스크를 반영한다.

### 로그 (Logging) — 15%

**근거**: 로그는 다른 영역의 위반을 "사후에 탐지하고 분석하는" 수단이다.

1. **실패 로그 (LOG-P0-008)**: 장애/보안 사고 발생 시 원인 추적의 유일한 수단이다. 로그 없이는 MTTR이 급격히 증가한다.

2. **감사 로그 (LOG-P0-010)**: 권한 변경 이력은 보안 감사의 핵심 근거이다.

**결론**: 로그는 그 자체로 보안/비용/권한 위반을 "방지"하지는 않지만, 위반 발생 시 "탐지와 대응"을 가능하게 하는 보조 영역이다. 직접적 위험 방지가 아닌 사후 대응 역할이므로 가장 낮은 15%를 부여한다.

## 등급 판정 기준

### 등급 체계

| 등급 | 점수 범위 | 조건 | 결정 | 필요 역할 승인 |
|------|-----------|------|------|---------------|
| **Green** | 90 ~ 100 | 모든 P0 PASS + 총점 90 이상 | 자동 배포 승인 | 없음 |
| **Yellow** | 75 ~ 89 | 모든 P0 PASS + 총점 75~89 | 조건부 승인 | PM 확인 필수 |
| **Red** | 0 ~ 74 | 총점 74 이하 **또는** P0 FAIL 1건 이상 | 배포 차단 | Engineer 수정 + 재검증 필수 |

### 핵심 원칙

1. **P0 절대 규칙**: P0 FAIL 1건이라도 존재 → 총점과 무관하게 Red 판정. 배포 자동 차단. 예외 없음.
2. **이진 판정**: 각 룰은 PASS(배점 100%) 또는 FAIL(0점) 이진 판정. 부분 점수 없음.
3. **N/A 처리**: 해당 기능 미사용 룰(예: BQ 미사용 시 COST-P0-007)은 분모에서 제외하고 총점을 정규화한다.
4. **P1 분리**: P1 룰은 P0 총점에 영향 없음. 별도 카운터로 추적하며, P1 위반 3건 이상 시 Warning 등급 부여.

### P1 위반 처리

| P1 위반 건수 | 처리 |
|-------------|------|
| 0건 | 이상 없음 |
| 1~2건 | 개선 권고 (리포트에 기록) |
| 3건 이상 | Warning 등급 승격 + 리뷰 필수 |

## 점수 계산 공식

### 기본 공식

```
영역 점수 = Σ(해당 영역 PASS 룰 배점)
총점 = 보안 점수 + 권한 점수 + 비용 점수 + 로그 점수
```

### N/A 정규화

N/A 룰이 있을 경우 해당 룰의 배점을 분모에서 제외하고 정규화:

```
정규화 총점 = (획득 점수 / (100 - N/A 룰 배점 합)) * 100
```

예시: COST-P0-007(5점)이 N/A인 프로젝트에서 나머지 룰 전부 PASS 시
```
획득 = 95, 분모 = 100 - 5 = 95
정규화 총점 = (95 / 95) * 100 = 100점 → Green
```

### 계산 예시

#### 예시 1: 완벽한 준수 (Green)

```
보안:  SEC-P0-005 PASS(18) + SEC-P0-004 PASS(12) + SEC-P0-009 PASS(10) = 40/40
권한:  AUTH-P0-003 PASS(10) + AUTH-P0-001 PASS(8) + AUTH-P0-002 PASS(7) = 25/25
비용:  COST-P0-006 PASS(10) + COST-P0-007 PASS(5) + COST-P0-011 PASS(5) = 20/20
로그:  LOG-P0-008 PASS(10) + LOG-P0-010 PASS(5) = 15/15
──────────────────────────────────
합계: 100/100 → Green (자동 배포 승인)
```

#### 예시 2: 비용 영역 미흡 (Yellow)

```
보안:  40/40 (전부 PASS)
권한:  25/25 (전부 PASS)
비용:  COST-P0-006 FAIL(0) + COST-P0-007 PASS(5) + COST-P0-011 PASS(5) = 10/20
로그:  15/15 (전부 PASS)
──────────────────────────────────
합계: 90/100 → Green (모든 P0 PASS? 아니요, COST-P0-006 FAIL)
P0 FAIL 1건 존재 → 총점 무관 → Red (배포 차단)
```

#### 예시 3: BQ 미사용 프로젝트 (N/A 정규화)

```
COST-P0-007: N/A (BQ 미사용)
나머지 10개 룰 전부 PASS: 획득 = 95점
정규화: (95 / 95) * 100 = 100점 → Green
```

## 커스텀 가중치 적용 방법

기본 가중치(Security 40% / Auth 25% / Cost 20% / Logging 15%)는 대부분의 AI 프로젝트에 적합하도록 설계되었다. 그러나 프로젝트 특성에 따라 가중치를 조정할 수 있다.

### 설정 방법

`compliance-config.yaml`의 `scoring.domain_weights` 필드를 수정한다:

```yaml
# compliance-config.yaml
scoring:
  domain_weights:
    security: 40    # 기본값: 40 (40%)
    auth: 25        # 기본값: 25 (25%)
    cost: 20        # 기본값: 20 (20%)
    logging: 15     # 기본값: 15 (15%)
```

### 제약 조건

1. **합계 100 필수**: 4개 영역의 가중치 합은 반드시 100이어야 한다. 합이 100이 아닌 경우 검증 시 에러를 출력한다.
2. **최소값 5**: 각 영역의 최소 가중치는 5이다. 특정 영역을 0으로 설정하여 완전히 무시하는 것은 허용하지 않는다.
3. **최대값 60**: 단일 영역의 최대 가중치는 60이다. 하나의 영역이 60%를 초과하면 다른 영역의 검증이 무의미해진다.
4. **룰별 배점 자동 재계산**: 영역 가중치 변경 시 해당 영역 내 룰별 배점은 영역 내 비중에 따라 자동으로 재계산된다.

### 커스텀 가중치 시나리오

#### 시나리오 A: 내부 전용 서비스 (보안 리스크 낮음)

외부 접근이 없는 내부 전용 서비스의 경우 보안 가중치를 낮추고 비용 관리를 강화할 수 있다.

```yaml
scoring:
  domain_weights:
    security: 25    # 내부 전용이므로 보안 리스크 상대적으로 낮음
    auth: 25        # 권한은 여전히 중요
    cost: 35        # 비용 최적화가 핵심 과제
    logging: 15     # 변동 없음
```

이 경우 룰별 배점 재계산:
```
SEC-P0-005: 25 * 0.45 = 11.25 → 11
SEC-P0-004: 25 * 0.30 = 7.5  → 8
SEC-P0-009: 25 * 0.25 = 6.25 → 6
                        합계 = 25
```

#### 시나리오 B: 금융/의료 서비스 (규제 중시)

규제 준수가 핵심인 도메인에서는 로그(감사 추적)와 보안을 강화한다.

```yaml
scoring:
  domain_weights:
    security: 40    # 보안 유지
    auth: 20        # 약간 감소
    cost: 10        # 비용보다 규제 준수 우선
    logging: 30     # 감사 로그 강화 (규제 요구사항)
```

#### 시나리오 C: MVP/프로토타입 (비용 집중)

빠른 출시가 목표이고 비용 관리가 최우선인 경우:

```yaml
scoring:
  domain_weights:
    security: 30    # 최소 보안 유지
    auth: 20        # 기본 권한 유지
    cost: 35        # 비용 관리 최우선
    logging: 15     # 변동 없음
```

### 룰별 배점 재계산 로직

영역 가중치가 변경되면 해당 영역 내 각 룰의 배점은 다음 공식으로 재계산된다:

```
룰 배점 = round(영역 가중치 * 룰의 영역 내 비중)
```

영역 내 비중(고정값):

| 영역 | Rule ID | 영역 내 비중 |
|------|---------|-------------|
| 보안 | SEC-P0-005 | 0.45 |
| 보안 | SEC-P0-004 | 0.30 |
| 보안 | SEC-P0-009 | 0.25 |
| 권한 | AUTH-P0-003 | 0.40 |
| 권한 | AUTH-P0-001 | 0.32 |
| 권한 | AUTH-P0-002 | 0.28 |
| 비용 | COST-P0-006 | 0.50 |
| 비용 | COST-P0-007 | 0.25 |
| 비용 | COST-P0-011 | 0.25 |
| 로그 | LOG-P0-008 | 0.67 |
| 로그 | LOG-P0-010 | 0.33 |

**반올림 보정**: 재계산 후 영역 내 합이 영역 가중치와 일치하지 않으면 가장 높은 비중의 룰에 보정값을 적용한다.

## 등급 임계값 커스터마이징

등급 판정 임계값도 프로젝트 상황에 따라 조정 가능하다:

```yaml
# compliance-config.yaml
gate:
  green_threshold: 90       # 기본값: 90 (90점 이상 → Green)
  yellow_threshold: 75      # 기본값: 75 (75~89점 → Yellow)
  p0_fail_override: true    # 기본값: true (P0 FAIL 시 무조건 Red)
```

### 제약 조건

1. `green_threshold` >= 80: Green 임계값을 80 미만으로 설정할 수 없다.
2. `yellow_threshold` >= 60: Yellow 임계값을 60 미만으로 설정할 수 없다.
3. `green_threshold` > `yellow_threshold`: Green은 항상 Yellow보다 높아야 한다.
4. `p0_fail_override`는 false로 설정할 수 없다: P0 절대 규칙은 해제 불가.

## 점수 개선 로드맵

### Phase 1: Red 탈출 (P0 전수 통과)

**우선순위**: 배점이 높은 룰부터 수정

1. SEC-P0-005 (18점) — Gateway 경유 강제. 가장 높은 배점이며 비용/로그 영역에도 긍정적 파급 효과
2. SEC-P0-004 (12점) — Firestore 직접 접근 제거. 서버 경유로 전환
3. AUTH-P0-003 (10점) — 서버 권한 미들웨어 추가. 전 API 일괄 적용 가능
4. LOG-P0-008 (10점) — error handler에 로그 호출 추가. 기존 코드에 삽입 가능
5. COST-P0-006 (10점) — AI 호출 경로에 비용 로그 추가. Gateway와 함께 적용 권장

### Phase 2: Yellow 유지 또는 Green 달성

- 나머지 P0 룰 수정 (배점 5~10점 구간)
- P1 위반 3건 미만으로 관리
- 재검증으로 총점 90+ 확인
